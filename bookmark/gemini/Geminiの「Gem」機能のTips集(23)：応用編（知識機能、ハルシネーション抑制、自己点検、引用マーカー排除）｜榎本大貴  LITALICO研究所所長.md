## 応用編

### 「知識」機能を使う

TIPSではなく標準機能の紹介です。なくても使えるので、意外と使っていない人がいるかなと思って念のため紹介します。

「知識」とは、AIに参照させる情報（ファイルやテキスト）のことです。

一方、「指示」（カスタム指示）は、AIに何をどうやってやらせるかを伝えるプロンプトのこと。

この2つの違いを具体例で説明すると：

- **指示** ：「議事録を作成してください。以下のフォーマットに従って出力してください」という「やり方」を伝えるもの
- **知識** ：実際の議事録のフォーマット例やテンプレートを保存しておく「材料」のようなもの

指示だけでも使えますが、知識に情報を保存しておくと、Gem本体を変更せずに知識だけ更新できるので便利です。

知識機能には、様々な情報を保存できます。例えば：

- **フォーマット・テンプレート** ：議事録やメールのフォーマット、よく使うテンプレート
- **専門用語集・用語辞典** ：業界用語の定義、社内で使う専門用語の説明
- **ルール・規約** ：会社のルール、業務手順書、ガイドライン
- **参照資料** ：過去の事例、ケーススタディ、参考になるドキュメント
- **データベース的な情報** ：顧客情報の形式、商品カテゴリの定義、分類基準

### TIPS11：知識を使って適用するフォーマットを変える

知識機能の強力な使い方の一つが、フォーマット例を保存しておくことです。

知識にGoogleドキュメントで作ったフォーマット例を保存しておき、それを参照させます：

```python
## 例1：フォーマット名を指定して参照
知識に保存されている「[フォーマット名]」を参照し、そのフォーマットに厳密に従って出力してください。

## 例2：知識ファイル名を直接指定
知識ファイル「議事録フォーマット.md」を参照し、その内容に従って出力してください。

## 例3：ユーザーの入力に応じて知識を選択
ユーザーが「フォーマットA」と指定した場合は、知識の「フォーマットA」を参照してください。
ユーザーが「フォーマットB」と指定した場合は、知識の「フォーマットB」を参照してください。
```

この方法のメリットは：

- 知識に複数のフォーマット例を保存しておける
- ユーザーが「フォーマットAで」と指定したら、知識のフォーマットAを参照させられる
- **知識の更新だけでフォーマットを変更できる** （Gem本体を変更する必要がない！）

最後のポイントが特に便利で、フォーマットを変更したい時にGem本体を変更しなくて済むんです。僕も、議事録のフォーマットを変更したい時に、知識ファイルだけ更新して対応しています。

### TIPS12：ハルシネーションを抑制する

ハルシネーションとは、AIが実際には存在しない情報を、あたかも存在するかのように作り出してしまうことです。  
『LLMとハルシネーション-基礎と対策-』（橘 秀幸, 稲原 宗能, 髙﨑 環, 福地 成彦）によると

> ① 文法的には正しく、言語としては大きな不自然さがない  
> ② 内容が事実に反していたり、論理的な矛盾がある  
> ③ 信じ込んでしまいそうなくらい説得力がある

『LLMとハルシネーション-基礎と対策-』

という特徴を持つものをハルシネーションとしています。

これは抑制したいですよね。AIが勝手に情報を補完してしまうのを抑制しましょう。

```python
以下のルールを厳守してください：
- 提供された情報のみを使用し、推測や補完は行わない
- 情報が不足している場合は、不足している旨を伝える
- 不確実な情報は「不明」と明記する
- ステップバイステップで考えて生成する
```

タスクの種類に応じて上記のルールは取り入れてみてください。

### TIPS13：情報源の明確化

出力する情報の出所を明確にしたい場合もありますよね。

```python
出力する情報について：
- 知識ファイルから取得した情報には「知識より」と明記
- ユーザーから提供された情報には「ご提供の情報より」と明記
- 推測や補完は行わず、事実のみを記載
```

これで、どの情報がどこから来たのかが分かるようになります。

### TIPS14：自己点検と反復修正

タスク実行後半に、自己点検と必要な修正を行うことで、最終的な出力品質を大幅に向上させられます。

例えば、指定したフォーマットや条件に完全に合致しているか、ミスや抜け漏れがないかを出力直前にチェックさせます。

```python
出力する前に、以下を必ず実施してください：
1. **要件に沿っているか自己点検**  
    - 指定されたフォーマット・条件・制約に完全に適合しているかチェックする  
    - ミスや抜け漏れがないか確認する

2. **必要に応じて再修正**  
    - もし自己点検で不備が見つかったら、必ず修正後の完成出力を再提示する

3. **冗長な前置きや説明を挿入しない**

**（命令例）下記ルールを絶対に遵守してください：**  
- 出力直前にマニュアルでの自己点検を必ず行い、不備があれば自動的に修正・再生成してください。
- チェック項目：内容の整合性／指定条件の厳守／余計な文章が混じっていないか
- 完成した出力例のみを提示し、チェック作業そのものは出力しないでください。
```

### TIPS15：引用・注釈マーカーの完全排除

AIが出力に引用マーカー（\[cite:\]みたいな記号）を入れてしまうことがあります。  
例えば下記のようにノイズであることを伝えたうえで最終出力の前に削除するように伝えます。

```ruby
**【重要：出力形式の厳格なクリーニング】**
Geminiの仕様上、ファイル参照時に \`[cite_end]\` といった引用タグが自動生成されることがありますが、本タスクにおいてこれらは**完全に不要なノイズ**です。
出力にあたっては、以下のルールを**最優先で**適用してください。

1. **システムタグの強制削除:**
   生成されたテキスト内に \`[cite\` または \`[source\` で始まるタグ（例：\`[cite:xxx]\`, \`[cite_end]\`など）が含まれていた場合、それらを**すべて空文字に置換（削除）**してから出力する。
   
2. **人間による清書として出力:**
   出力結果は、人間が手書きで清書したかのように、システム的な注釈記号が一切ない「純粋なテキスト」に仕上げる。
   
3. **脚注・参照の禁止:**
   文末や文中に参照元のファイル番号やページ数を記載しない。
```

### TIPS16：出力の段階的表示

処理が複数ステップに分かれる場合、進捗を表示すると良いです。

```python
処理が複数ステップに分かれる場合：
1. 各ステップの完了を報告
2. 次のステップに進む前に確認を取る
3. 全体の進捗を表示
```

これで、ユーザーが処理の進捗を把握できますし、途中で思っていた出力と異なる場合に、軌道修正することができます。
